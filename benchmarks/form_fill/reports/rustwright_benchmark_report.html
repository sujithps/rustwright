01

Methodology

The harness separates client-library work from browser hosting and excludes page-load time from the latency comparisons.

Remote execution

Both clients drive remote Skyvern cloud browsers over CDP. Browser memory is off-box, so the memory result isolates the local client process tree.

Navigation exclusion

Website and network time are excluded as a navigation band rather than attributed to either library.

Two evidence lanes

Controlled data: URL fixtures provide byte-identical work and zero render; live automation-sandbox cases exercise real navigation and interaction.

Library probes

Page-wait-free probes isolate CDP and library pipelines. Controlled click cases set page_wait=0.

03

Concentrated weaknesses

The smoke suite exposes specific implementation paths, not a general across-the-board slowdown.

W1 CRITICAL

query_selector_all handle-returning path

~270 ms/element: N=10 → 2.8s, N=50 → 13.5s, and N=100 → 27s. Playwright is ~1 ms/element, including N=1000 → 875 ms.

locator.count() is fine and remains flat at ~150 ms, localizing the regression to the handle-returning path.

Cause: one CDP round-trip per ElementHandle, with no batching. Fix: batch handle materialization.

W2 HIGH

Object-evaluate return path

Small-object evaluation is ~2.5x slower: Rustwright 138 ms vs Playwright 55 ms.

Target: profile serialization and remote-object conversion on the object-return path.

W3 HIGH

Initial CDP connection

Connect is ~3x slower in the corrected smoke result: Rustwright 1336 ms vs Playwright 449 ms.

Likely mechanism: eager target attach plus Tokio and event-stream setup. Confirm with higher-repetition instrumentation before optimizing.

These are actionable optimization targets for the team: batch handle creation first, then profile object conversion and connection initialization.

04

Memory

The strongest architectural finding is the local client footprint. Remote browser memory remains off-box and outside this measurement.

Client memory

45 vs 126 MB

Client process-tree RSS with the browser remote: Rustwright uses ~2.8x less, or 65% less memory.

Architectural basis

in-process Rust

The result is consistent with an in-process Rust client versus Python plus a Node driver, and matches the repository’s known 66%-less-memory claim.

06

Caveats and next evidence

Read the direction of the controlled results, but do not treat this smoke stage as a publication-grade confidence interval.

  • Memory is macOS process-tree RSS, not proportional set size (PSS).
  • Repetition counts are low; variance and tail latency are not characterized.
  • Single-sample connect variance was corrected during testing: an earlier “faster connect” result was a fluke.
  • The two clients were compared in separate remote browser sessions, not one shared session.
  • Precise PSS, high-repetition statistics, and complete provenance require the Blacksmith run.
